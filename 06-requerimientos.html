CAPÍTULO 06

Requerimientos

El 72% de los proyectos falla — y los requisitos mal gestionados son la causa #1.

¿Qué es un requerimiento?

Pfleeger: "Una característica del sistema o una descripción de algo que el sistema es capaz de hacer con el objeto de satisfacer el propósito del sistema."

La IEEE 610-1990 lo define como:

  1. Condición o capacidad que necesita el usuario para resolver un problema.
  2. Condición o capacidad que debe satisfacer un sistema o componente para cumplir un contrato, estándar o especificación.

Causas principales de falla

  • Requisitos deficientes, incompletos o cambiantes.
  • Planificación sin base en los requisitos.
  • Estimaciones poco realistas.
  • Falta de participación del usuario.

Dominio: aplicación vs. software

Dominio de la aplicación

Donde existe el requerimiento del cliente. El qué hay que hacer.

Dominio del software

Provee la solución. El cómo se construye.

El problema de la abstracción

El software es difícil de visualizar. Es fácil que el cliente diga "esto no es lo que pedí" aunque pensaba que sí lo era.

Recomendaciones

  • No quedarse con descripciones genéricas — ahondar en detalles.
  • Evitar ambigüedades.
  • No dar nada por supuesto.
  • Dedicar tiempo a la especificación.

Defectos comunes

Falta de involucramiento del cliente/usuarios

Sin un interlocutor presente, cualquier cosa puede pasar.

Cambios sin gestión del impacto

Todo cambio de requisito afecta tiempo, costo y otros requerimientos.

Requisitos ambiguos

Dos interpretaciones posibles del mismo requerimiento.

Requisitos innecesarios

El desarrollador agrega cosas que nadie pidió.

Requisitos insuficientes

Definición superficial.

Omisión de grupos de usuarios

Usuarios con necesidades específicas olvidados.

Tipos de requerimientos

Describen QUÉ hace el sistema. Entradas, salidas, interacciones, comportamiento ante cada estímulo.

Si el sistema registra pedidos, el requerimiento debe especificar qué datos se ingresan, qué controles se hacen y qué salida se genera — tanto para válidos como inválidos.

Describen RESTRICCIONES sobre el sistema. Limitan las opciones de diseño.

  • Performance, seguridad, portabilidad, eficiencia.
  • Plataforma o lenguaje de programación a usar.
Los requerimientos llegan en lenguaje natural → ambigüedad → hay pérdida de información en la transformación a la especificación formal.

Beneficios de buenos requisitos

🤝 Acuerdo entre devs, usuarios y cliente.

📊 Base objetiva para estimar recursos.

✅ Permite validar que el producto cumple.

💰 Reduce costos — evita re-correcciones.
