CAPÍTULO 06

Requerimientos

El 72% de los proyectos falla — y los requisitos mal gestionados son la causa #1.

¿Qué es un requerimiento?

Pfleeger: "Una característica del sistema o una descripción de algo que el sistema es capaz de hacer con el objeto de satisfacer el propósito del sistema."

La IEEE 610-1990 lo define como:

  1. Condición o capacidad que necesita el usuario para resolver un problema.
  2. Condición o capacidad que debe satisfacer un sistema o componente para cumplir un contrato, estándar o especificación.

Dominio: aplicación vs. software

Dominio de la aplicación

Donde existe el requerimiento del cliente. El qué hay que hacer.

Dominio del software

Provee la solución. El cómo se construye.

El problema de la abstracción

El software es difícil de visualizar. Es fácil que el cliente diga "esto no es lo que pedí" aunque pensaba que sí lo era.

Recomendaciones

  • No quedarse con descripciones genéricas — ahondar en detalles.
  • Evitar ambigüedades.
  • No dar nada por supuesto.
  • Dedicar tiempo a la especificación.

Defectos comunes

Falta de involucramiento del cliente/usuarios

Sin un interlocutor presente, cualquier cosa puede pasar.

Cambios sin gestión del impacto

Todo cambio de requisito afecta tiempo, costo y otros requerimientos.

Requisitos ambiguos

Dos interpretaciones posibles del mismo requerimiento.

Requisitos innecesarios

El desarrollador agrega cosas que nadie pidió.

Requisitos insuficientes

Definición superficial.

Omisión de grupos de usuarios

Usuarios con necesidades específicas olvidados.

Tipos de requerimientos

Describen QUÉ hace el sistema. Entradas, salidas, interacciones, comportamiento ante cada estímulo.

Si el sistema registra pedidos, el requerimiento debe especificar qué datos se ingresan, qué controles se hacen y qué salida se genera — tanto para válidos como inválidos.

Describen RESTRICCIONES sobre el sistema. Limitan las opciones de diseño.

  • Performance, seguridad, portabilidad, eficiencia.
  • Plataforma o lenguaje de programación a usar.
Los requerimientos llegan en lenguaje natural → ambigüedad → hay pérdida de información en la transformación a la especificación formal.

Beneficios de buenos requisitos

🤝 Acuerdo entre devs, usuarios y cliente.

📊 Base objetiva para estimar recursos.

✅ Permite validar que el producto cumple.

💰 Reduce costos — evita re-correcciones.

IEEE 610-1990:

  1. Condición o capacidad que necesita el usuario para resolver un problema.
  2. Condición o capacidad que debe satisfacer un sistema o componente para cumplir un contrato, estándar o especificación.

Es la base del contrato entre cliente y equipo.

Funcional vs No funcional

Funcional: describe QUÉ hace el sistema — entradas, salidas, comportamiento ante cada estímulo, casos válidos e inválidos.

Ej: "el sistema debe registrar pedidos validando stock antes de confirmar."

No funcional: describe RESTRICCIONES sobre el sistema o sobre cómo se desarrolla — performance, seguridad, portabilidad, lenguaje, plataforma.

Ej: "el sistema debe responder en < 2 s", "debe usar PostgreSQL".

Ingeniería de Requerimientos — 4 etapas
  1. Extracción: obtener información del cliente y usuarios (entrevistas, cuestionarios, observación).
  2. Análisis: procesar la información, detectar ambigüedades, conflictos y omisiones.
  3. Especificación: documentar formalmente en el SRS.
  4. Validación: el cliente confirma que lo especificado refleja lo que quiere.

Es iterativa: los cambios vuelven a extracción.

Costo de los errores por etapa

Un error cuesta exponencialmente más mientras más tarde se detecta:

  • En requerimientos: costo base (×1).
  • En diseño: ×5 a ×10.
  • En codificación: ×20.
  • En pruebas: ×50.
  • En producción: hasta ×5000.

Por eso la inversión en una buena IR paga solo.

Standish Group — números del fracaso

Estudio sobre 280.000 proyectos (año 2000):

  • 28% exitosos (a tiempo, presupuesto y alcance).
  • El resto se pasa de tiempo/costo o se cancela.
  • Causa #1 de fracaso: requisitos deficientes, incompletos o cambiantes.

Fundamenta la importancia de la IR.

Defectos comunes en los requerimientos
  • Falta de involucramiento del cliente.
  • Cambios sin gestión del impacto.
  • Ambigüedades (dos interpretaciones posibles).
  • Requisitos innecesarios (los agrega el dev).
  • Requisitos insuficientes (descripción superficial).
  • Omisión de grupos de usuarios.

Preguntas tipo parcial

P6 · ¿Qué es un requerimiento? Explique las diferencias entre requerimiento funcional y no funcional. (2 puntos)

Requerimiento

Según Pfleeger, un requerimiento es "una característica del sistema o una descripción de algo que el sistema es capaz de hacer con el objeto de satisfacer el propósito del sistema".

La IEEE 610-1990 lo define como:

  1. Condición o capacidad que necesita un usuario para resolver un problema o alcanzar un objetivo.
  2. Condición o capacidad que debe satisfacer un sistema o componente para cumplir un contrato, estándar o especificación.

Es la base del contrato entre cliente y equipo, y entrada de todo el proceso de desarrollo. Un requerimiento debe ser preferentemente correcto, completo, verificable, no ambiguo, consistente y rastreable.

Requerimiento funcional

Describe qué debe hacer el sistema: las funciones, servicios o comportamientos observables. Especifica entradas, salidas, interacciones y cómo se comporta frente a cada estímulo (incluyendo casos inválidos o de error).

Ejemplos:

  • "El sistema debe permitir al usuario registrar un pedido indicando productos, cantidades y dirección de entrega."
  • "Al confirmar un pedido, el sistema debe validar el stock y emitir un comprobante."

Requerimiento no funcional

Describe restricciones sobre el sistema o sobre cómo se construye. No cambia qué hace el sistema, pero limita las opciones de diseño e implementación.

Incluye: performance, seguridad, disponibilidad, portabilidad, usabilidad, interoperabilidad, así como restricciones de plataforma, lenguaje o normativa.

Ejemplos:

  • "El tiempo de respuesta a cualquier consulta debe ser menor a 2 segundos con 1000 usuarios concurrentes."
  • "El sistema debe desarrollarse en Java y desplegarse sobre PostgreSQL."
