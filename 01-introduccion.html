CAPÍTULO 01

Introducción a la ingeniería de software

De dónde viene la disciplina, qué es un producto, cualidades, principios y la tesis de Brooks.

¿De dónde viene esto?

En 1968, la OTAN organizó la primera conferencia sobre desarrollo de software. Ahí nacieron dos términos que siguen vigentes:

Crisis del software

Proyectos que se pasaban de costo y tiempo, o directamente se cancelaban. Los que se completaban tenían mantenimiento carísimo.

Ingeniería del software

La disciplina que busca resolver la crisis aplicando métodos, técnicas y herramientas sistemáticas.

Las 3 definiciones clave

"Establecimiento y uso de principios de ingeniería para obtener software económico que trabaje de forma eficiente en máquinas reales."
"La aplicación de métodos sistemáticos, disciplinados y cuantificables para el desarrollo, operación y mantenimiento de software."
"La aplicación práctica del conocimiento científico al diseño y construcción de programas y la documentación asociada requerida para desarrollarlos, operarlos y mantenerlos."

Producto ≠ solo el ejecutable. Incluye el código, la documentación de requerimientos, diseño, datos de prueba, manuales. Todo.

Sistemas vs. software

Un sistema (IEEE 610.12) es una colección de componentes organizados para cumplir una función. Los componentes son:

💻 Software
🔩 Hardware
👥 Gente
🗄️ Base de datos
📄 Documentación
📋 Procedimientos

El software es uno de esos componentes — lógico, no físico. Puede ser un sistema independiente o un subsistema. Ej: el GPS del celular es un subsistema del software del celular.

Principios de la ingeniería de software

Rigor y formalidad

Complemento a la creatividad. El nivel máximo es la formalidad matemática.

Separación de incumbencias

Dividir el problema en aspectos manejables por separado (equipos de dev, testing, redes).

Modularidad

Dividir el sistema en módulos con alta cohesión (cada módulo hace una cosa bien) y bajo acoplamiento (los módulos dependen poco entre sí).

Abstracción

Identificar lo importante e ignorar lo irrelevante.

Anticipación al cambio

Prever que el software se va a actualizar. Las herramientas de control de versiones ayudan.

Generalización

Buscar el problema más general detrás del problema puntual.

Incrementalidad

Entregar subconjuntos funcionales para feedback temprano.

Flashcards

¿Qué es la Ingeniería de Software?

Disciplina que aplica métodos, técnicas y herramientas sistemáticas al desarrollo, operación y mantenimiento de software.

Surge en la conferencia de la OTAN de 1968 como respuesta a la "crisis del software".

Definiciones:

  • Bauer (1968): principios de ingeniería para obtener SW económico y eficiente.
  • IEEE (1993): aplicación de métodos sistemáticos, disciplinados y cuantificables.
  • Boehm: aplicación práctica del conocimiento científico al diseño y construcción de programas.
Principios: Rigor y Formalidad

Rigor: enfoque preciso y disciplinado que complementa (no reemplaza) a la creatividad del desarrollo.

Formalidad: nivel máximo de rigor — uso de notaciones matemáticas que permiten razonar y verificar el software de forma no ambigua.

Ambos reducen la subjetividad y facilitan la verificación, documentación y comunicación dentro del equipo.

Brooks — Esencia y Accidente

En "No Silver Bullet" Brooks divide las dificultades del software en dos:

Esenciales (inherentes, no desaparecen):

  • Complejidad: cada SW es único, con miles de estados.
  • Conformidad: debe adaptarse a interfaces humanas e institucionales.
  • Variabilidad: está sometido a cambios constantes.
  • Invisibilidad: no se puede visualizar fácilmente.

Accidentales: las que se resuelven con tecnología (lenguajes pobres, poca memoria, herramientas débiles).

Producto de software y Modularidad

Producto ≠ ejecutable. Incluye código fuente, documentación de requisitos y diseño, datos de prueba y manuales — todo.

Modularidad = dividir el sistema en módulos con:

  • Alta cohesión: cada módulo hace una sola cosa bien.
  • Bajo acoplamiento: los módulos dependen poco entre sí.

Facilita mantenibilidad, verificabilidad y reuso.

Cualidades internas vs externas (Ghezzi)

Internas: interesan al desarrollador — verificabilidad, mantenibilidad, reusabilidad, portabilidad, productividad.

Externas: visibles al usuario — correctitud, confiabilidad, robustez, performance, amigabilidad, integridad.

Las internas habilitan las externas: un SW mantenible es más fácil de volver confiable.

Productividad (cualidad)

Cualidad interna del proceso de software: cantidad de producto útil entregado por unidad de recurso (tiempo, personas, dinero).

Mejora con: reuso, buenas herramientas, automatización de pruebas, procesos definidos y equipos cohesionados.

No confundir con performance (cualidad del producto, no del proceso).

Preguntas tipo parcial

P1 · ¿Qué es la Ingeniería de Software? (1 punto)

La Ingeniería de Software es la disciplina que se ocupa del desarrollo, operación y mantenimiento del software aplicando métodos, técnicas y herramientas sistemáticas, disciplinadas y cuantificables (IEEE, 1993). Nace en la conferencia de la OTAN de 1968 como respuesta a la crisis del software (proyectos con sobrecosto, atrasos y mantenimiento caro).

Otras definiciones clave:

  • Bauer (1968): "Establecimiento y uso de principios de ingeniería para obtener software económico que trabaje de forma eficiente en máquinas reales."
  • Boehm: "Aplicación práctica del conocimiento científico al diseño y construcción de programas y la documentación asociada requerida para desarrollarlos, operarlos y mantenerlos."

No se limita al código: abarca el producto completo (código, documentación, pruebas, manuales) y el proceso que lo genera.

P3 · Explique la cualidad de Productividad y los principios de Rigor y Formalidad. (2 puntos)

Productividad

Es una cualidad interna del proceso (no del producto) definida por Ghezzi. Mide la cantidad de producto útil entregado por unidad de recurso consumido (tiempo, personas, costo). Un proceso productivo permite obtener un software de calidad con el menor costo posible. Se mejora con reuso, automatización, buenas herramientas y equipos cohesionados.

Rigor

Es uno de los principios de la ingeniería de software: enfoque preciso y disciplinado que complementa (no reemplaza) a la creatividad. Impone ser sistemáticos al analizar, diseñar y documentar. Reduce la subjetividad y los errores.

Formalidad

Es el grado máximo de rigor: uso de notaciones matemáticas precisas y no ambiguas para describir o verificar el software. Permite demostrar propiedades del sistema (p. ej. correctitud) sin depender de la interpretación humana.

Relación: la formalidad es siempre rigurosa, pero el rigor no exige necesariamente formalidad.

P4 · Según Brooks ("No Silver Bullet"): dificultades esenciales vs. accidentales. Enumere y explique las esenciales. (2 puntos)

Frederick Brooks, en "No Silver Bullet: Essence and Accidents of Software Engineering", sostiene que no existe una única técnica o herramienta que resuelva por sí sola los problemas del software. Para explicarlo divide las dificultades en dos categorías:

Dificultades accidentales

Las resolubles con avances tecnológicos: lenguajes limitados, poca memoria, herramientas débiles, compiladores lentos. Son temporales — la tecnología las va mitigando.

Dificultades esenciales (inherentes, no desaparecen)

  1. Complejidad: cada sistema software es único y contiene muchos estados posibles que interactúan entre sí. A diferencia de otras disciplinas, los componentes no se repiten — si dos funciones son idénticas, se fusionan. El resultado es que todo software nuevo tiene una complejidad irreducible.
  2. Conformidad: el software debe adaptarse a interfaces humanas, institucionales, legales y a otros sistemas preexistentes. Esa conformidad es arbitraria y externa al problema, por lo que es imposible eliminarla con abstracción.
  3. Variabilidad: el software está sometido a cambios constantes — nuevas necesidades, regulaciones, entornos. Un software exitoso se modifica toda su vida útil.
  4. Invisibilidad: el software es intangible, no se puede visualizar geométricamente como un plano o una máquina. Esto dificulta la comunicación, el diseño y la detección de fallas.

Conclusión: los avances resuelven las accidentales, pero las esenciales requieren buenas prácticas de ingeniería (modularidad, abstracción, reuso, prototipado, equipos hábiles) en vez de una "bala de plata".

🔑 Ideas clave para el parcial

  • Las 3 definiciones: Bauer, IEEE, Boehm.
  • Producto ≠ solo ejecutable — incluye toda la documentación.
  • Cualidades internas vs. externas (Ghezzi).
  • Modularidad: alta cohesión, bajo acoplamiento.
  • Esencia vs. accidente (Brooks): las esenciales nunca desaparecen.
